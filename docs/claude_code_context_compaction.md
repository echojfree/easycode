# 你的 AI 编程助手是怎么"记住"一整天对话的？

> 深扒 Claude Code 源码，看顶级 AI 工具如何优雅地处理"记忆溢出"

---

## 先说一个你一定遇到过的问题

用 AI 写代码，聊着聊着它突然"失忆"了。

刚才还在帮你调试的那个函数，问它一句"上面那个 bug 解决了吗"，它一脸懵逼。或者更糟——它开始胡说八道，把前面的上下文全混了。

这不是 AI 变笨了，而是它的"短期记忆"装满了。

---

## LLM 的记忆是有上限的

每个大模型都有一个**上下文窗口**（Context Window）——你可以理解为它的工作台大小。

Claude 的工作台是 **200,000 个 token**（大约 15 万个汉字）。听起来很大，但你想想：一个下午的编程会话，动辄几十次来回，每次还要附上文件内容、报错信息、工具执行结果……工作台很快就堆满了。

**工作台满了怎么办？**

普通的做法：直接报错，让用户自己开个新对话。

Claude Code 的做法：**悄悄收拾桌子，但不打断你的工作。**

---

## Claude Code 的四层"桌面收拾术"

我最近翻了 Claude Code 的源码（在 `src/services/compact/` 目录下），发现它有一套精心设计的四层处理机制，从轻到重依次出手。

### 第一层：裁剪工具结果——最轻量的处理

你每次让 Claude 读一个文件、执行一条命令，结果会塞进对话历史里。

一个大文件读下来可能几千行，早期读过的文件内容往往后面用不到了。Claude Code 的做法很简单粗暴：**把超长的工具结果头尾各保留一段，中间用省略号替代。**

```
原内容: [12000 字节的文件内容]

压缩后:
[tool result compacted] (original=12000 chars)
前 6000 字节...
... (snipped) ...
后 2000 字节
```

不调用 AI，纯规则处理，成本为零，几乎无感知。

---

### 第二层：微压缩——利用服务端缓存

这一层更隐蔽。

Claude Code 会记录你上次和 AI 对话的时间。如果**超过 60 分钟没有对话**，它会认为服务器端的 KV 缓存已经过期，然后主动把历史消息中的工具调用内容清空，只保留最近 5 条工具记录。

道理很简单：过了一个小时你都没用的中间结果，大概率以后也用不到。清掉它，腾出空间给新的工作。

---

### 第三层：自动压缩——让 AI 给自己写摘要

这是最有意思的一层。

**触发条件**：当 token 用量达到 187,000（200K 窗口减去 13,000 的缓冲），自动触发。

**做什么**：Claude Code 会调用 Claude 自己，让它对之前的全部对话生成一份结构化摘要。

是的，你没看错——**用 AI 来压缩 AI 的记忆**。

生成的摘要包含 9 个维度：

```
1. 用户想解决的核心问题
2. 涉及的技术概念
3. 修改过哪些文件
4. 遇到了哪些错误
5. 如何解决的
6. 用户提出了哪些关键要求
7. 还有哪些 TODO 未完成
8. 当前正在做什么
9. 接下来应该做什么
```

压缩完成后，之前几十条对话变成一条摘要消息，Claude 的工作台重新变得整洁，但它"记得"之前做了什么。

```
[之前 180 条消息] ──压缩──> [1 条摘要] + [最近 10 条原始消息]
```

用户几乎感知不到切换，对话可以无缝继续。

---

### 第四层：硬限制——最后的防线

如果以上三层都没能及时处理，token 继续涨到 197,000，Claude Code 会**直接拒绝这条请求**，返回错误信息，提示用户手动压缩或开新对话。

这是一道安全阀，防止 AI 在上下文极度拥挤的情况下输出胡言乱语。

## 一张图理解整个流程

```
每次发消息前，Claude Code 内部执行：

你发消息
    ↓
① 截断超长工具结果（纯规则，无感）
    ↓
② 检查距上次对话是否超过 60 分钟？
   是 → 清空旧工具记录（微压缩）
    ↓
③ Token 用量 > 187,000？
   是 → 调用 AI 生成摘要（自动压缩）
    ↓
④ Token 用量 > 197,000？
   是 → 报错拒绝请求
    ↓
正常调用 Claude API
```

---

## 这套设计聪明在哪里

**1. 成本分级。** 能用规则解决的绝不调 LLM。第一层、第二层成本接近零，只有真的撑不住了才动用 AI 生成摘要。

**2. 摘要质量有保障。** 让 AI 总结 AI 自己的工作，比人工规则提取信息要准确得多。AI 知道哪些是重点，哪些是过程细节。

**3. 保留最近的原始消息。** 压缩不是全部替换，而是"旧的摘要化 + 新的原样保留"。最近的对话上下文最重要，绝对不压缩，保证 AI 回复的连贯性。

**4. 断路器防止雪崩。** 如果压缩本身连续失败 3 次（比如 AI 摘要接口异常），断路器触发，停止尝试，防止把 token 都浪费在压缩失败上。

---

## 对我们自己开发 AI 应用有什么启示

如果你也在开发带有长对话的 AI 应用，Claude Code 这套分层策略值得借鉴：

- **不要只靠截断。** 简单把旧消息删掉会丢失重要上下文，导致 AI"失忆"。
- **摘要比删除好。** 哪怕是简单的规则摘要，也比直接清空强。
- **工具结果是大头。** 对话本身通常不大，撑爆上下文的往往是 `read_file`、`bash` 这类工具返回的大量内容。优先压缩它们。
- **让 AI 摘要自己的历史。** 如果你的场景允许，这是目前质量最高的做法。
